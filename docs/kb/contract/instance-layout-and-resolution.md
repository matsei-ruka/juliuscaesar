## Summary

Every `jc-*` tool operates on an instance directory. Resolution order is consistently documented as:

1. `--instance-dir <path>`
2. `$JC_INSTANCE_DIR`
3. Walk upward from cwd for a `.jc` marker
4. Tool-specific cwd fallback, such as `memory/`, `heartbeat/tasks.yaml`, `voice/`, or `ops/`

## Instance scaffold

`jc init` copies `templates/init-instance/` into a target, writes a `.jc` marker, creates `.env` with mode 600, and refuses to scaffold over an existing instance or a non-empty target directory except allowed root files.

Expected base layout:

- `.jc`
- `.env`
- `.gitignore`
- `CLAUDE.md`
- `memory/L1`
- `memory/L2`
- `memory/raw`
- `heartbeat/tasks.yaml`
- `heartbeat/fetch`
- `voice/references`
- `voice/tmp`
- `ops/watchdog.conf`
- `.codex/`

## Contracts

- `.jc` is the marker for walk-up auto-discovery.
- `.env` is the only expected secret file and should not be committed.
- `CLAUDE.md` imports L1 memory files into Claude Code sessions.
- `memory/INDEX.md` is generated by `jc memory rebuild`.
- `heartbeat/state`, `voice/tmp`, memory SQLite files, and worker state are runtime or derived state.

## Invariants

- Tools take explicit instance paths internally; libraries should avoid global instance state.
- Framework code does not depend on a particular user instance.
- Instance layout is intentionally Obsidian-compatible for memory.
